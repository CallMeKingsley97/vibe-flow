# ADR-016：SwiftUI 风格设计语言与令牌化

- 状态：Accepted
- 日期：2026-07-17

## 背景

Vibe Flow 是一款桌面级本地观察工具，用户主要是重度使用 macOS 与终端的开发者。此前界面使用即兴的深色 UI，颜色、圆角、字号散落在各处的 CSS 中，随着 M7（跨维度洞察）和后续里程碑陆续加入更多复杂视图，视觉体系无法继续按"手动调色"的方式演进。

用户在 M7 启动时明确要求：新加任何功能需要 UI 美观，遵循 Apple SwiftUI 的设计美学。这需要一套明确的规范，并把规范固化到工程结构中。

## 决策

1. 采用 SwiftUI/macOS 系统应用的视觉基调：语义色、清晰层级、克制装饰、圆角与留白遵循 4px 网格与 Apple 常见比例。
2. 在 `src/app/styles.css` 顶部集中声明 CSS 变量作为唯一的设计令牌来源，覆盖颜色、圆角、间距、字号、字重、材质、阴影、动效时长与曲线。
3. 在 `docs/DESIGN_LANGUAGE.md` 中固化设计规范，包含组件规则、布局节律、视觉验收流程和变更流程。
4. 所有新增视图必须消费令牌，不允许写死颜色/圆角/间距/字号；既有代码在改动时应逐步迁移。
5. 强调色（tint）保持既有的青绿系，与品牌 mark 一致；语义色（critical、warning、caution、positive）只在真实语义下使用，不作装饰。
6. 视觉验收作为质量门：新增视图需要在深/浅主题各截一张 1440px 截图，并在 PR 中回顾"基调是否符合"。

## 后果

- 优点：视图能够快速拓展、跨页面视觉一致；主题切换只需要维护令牌；未来若加入其他强调色或系统主题（自动 tint 变化）只需扩展令牌。
- 代价：需要一次性重写 `styles.css` 顶部；已有硬编码颜色的代码需要在触碰时迁移；开发者需要熟悉令牌命名。
- 风险：令牌爆炸。缓解方式是把语义清单固化在 DESIGN_LANGUAGE.md，新增令牌要与既有语义对齐。

## 落地范围

- 新建 `docs/DESIGN_LANGUAGE.md`；
- 在 `src/app/styles.css` 顶部添加设计令牌区，并把 M7 洞察视图作为首个完全消费令牌的模块；
- 在 `project-architecture.md` 与 Roadmap 中补充设计语言的引用。

## 备选方案

- 引入 Tailwind / CSS 框架：会带来样式重写和构建复杂度，Vibe Flow 不需要覆盖 Web 场景，暂不采用；
- 使用 Radix / Ariakit 组件库：Vibe Flow 组件数量少、交互轻，引入依赖成本高；先以自建令牌+组件为主。
